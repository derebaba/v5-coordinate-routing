Fieldwork Scheduler: Data Flow Map

This one-page guide explains how the app is structured, what each page controls, and how school, planning, route, and verification data update each other. It is written for daily use by the fieldwork team, not only for technical maintenance.

4. Data Update Rules

Action What gets updated What you see change next
Import schools in City Setup The current city's school database is created or extended. Schools list, planner dropdowns, progress table, route builder school set.
Edit school details in Operations The school record itself changes. School table, planner row details, printed day report, progress grouping, route builder readiness.
Edit school coordinates The school's `latitude/longitude` values change and cached routes for that school are cleared. Route hints may disappear until the route cache is rebuilt or re-imported.
Set manual school status The school's manual status override becomes the first status source. Status labels, school row colors, progress counts, planner availability behavior.
Save a day plan Assignments move from draft into the city database. School status can move to scheduled, remaining classrooms shrink, progress numbers increase.
Save day verification The city's verification records are updated for that day and those schools. School status becomes completed or incomplete, remaining classrooms and future scheduling update.
Import route cache Stored drive times for school pairs are updated for the city. Planner route hints become available without a live API connection.

5. Planner Logic

Daily Planning

  1. Choose a day tab.
  2. Available researchers appear as rows.
  3. Select district first, then school.
  4. The school row auto-fills working hours, lunch break, duration, max class size, and route hints.
  5. Remaining classroom counts update live across all researchers in the same day.
  6. Once saved, the day locks until `Edit` is used.

Capacity and Tablet Rules

  • A school disappears from the dropdown when its remaining classrooms reach zero.
  • Hard classroom capacity cannot be saved above the school's total classroom count.
  • Required tablets are computed per researcher, then the maximum relevant value is used for that researcher.
  • If `Sarmal` is checked for a row, that row's max class size is doubled before comparing it against the researcher's other assigned schools.

6. Status Hierarchy

Completed Highest final state. School is treated as fully done. It should not need future scheduling.
Incomplete Some classrooms still remain. The school can appear again in future planning.
Scheduled The school appears in saved day plans, but has not yet been marked completed through verification or manual status.
Not scheduled No saved assignments yet, or a manual override reset the school back to unscheduled.

If a manual status is set in the school editor, it overrides the automatic status calculation. If the status is left as `Automatic`, the app computes it from planning and verification data.

7. Route Cache Flow

  1. School file includes `latitude` and `longitude`.
  2. Distance Cache builder calculates or imports school-pair routes.
  3. Those route rows are stored per city.
  4. The planner reads cached routes only. It does not need a live API connection during normal use.
Coordinates are the source of truth Changing coordinates clears route rows for that school Rebuild cache after coordinate edits

8. Practical Team Workflow

  1. Create or select the city tab.
  2. Import school Excel in City Setup.
  3. Build or import route cache if drive times are needed.
  4. Add working days and researchers.
  5. Use Operations daily for planning, printing, and end-of-day verification.
  6. Edit school records from the schools list whenever details, coordinates, or status need manual correction.
  7. Export JSON regularly as a backup of all cities.

# Phase 3.1: Editable Sharing - Research

**Researched:** 2026-03-21
**Domain:** Full-stack feature — ASP.NET Core route + vanilla JS frontend state management
**Confidence:** HIGH

## Summary

This phase adds an "editable share" mode: a URL like `/edit/{id}` that loads the full scheduler app (index.html + app.js) with the shared document's data pre-loaded for editing. Users can modify data and save changes back to the server via PUT to the same document ID.

The implementation is straightforward because all required API endpoints already exist (GET `/documents/{id}`, PUT `/documents/{id}`), the frontend already has battle-tested patterns for loading external data (`normalizeState` + `replaceState`), and the viewer route in Program.cs provides a template for the new `/edit/{id}` route. The main complexity lies in (a) serving the frontend files from the API server (currently they're only used as local files), (b) handling relative asset paths when served from a sub-path, and (c) wiring up edit-mode detection and save-back flow in app.js.

**Primary recommendation:** Mirror the `/viewer/{id}` route pattern to serve index.html, copy frontend assets into wwwroot during Docker build, inject a `<base href="/">` tag to fix relative paths, and add a thin edit-mode layer in app.js that auto-fetches document data and switches upload from POST to PUT.

## Architecture Patterns

### Recommended Approach

#### 1. Backend: `/edit/{id}` Route (Program.cs)
Mirror the existing `/viewer/{id}` route pattern exactly:

```csharp
// Source: Program.cs lines 184-192 (existing viewer pattern)
app.MapGet("/edit/{id}", async (string id, IWebHostEnvironment env) =>
{
    var filePath = Path.Combine(env.WebRootPath, "index.html");
    if (!File.Exists(filePath))
        return Results.NotFound();
    var html = await File.ReadAllTextAsync(filePath);
    return Results.Content(html, "text/html");
});
```

**Key decision — serving index.html with correct asset paths:**

The current `index.html` uses **relative paths**: `href="styles.css"` and `src="app.js"`. When served from `/edit/some-uuid`, the browser resolves these to `/edit/styles.css` and `/edit/app.js` — which don't exist.

**Solution: Inject `<base href="/">`** into the HTML before serving. This tells the browser to resolve all relative URLs from the root path, where static files are served from `wwwroot/`.

```csharp
app.MapGet("/edit/{id}", async (string id, IWebHostEnvironment env) =>
{
    var filePath = Path.Combine(env.WebRootPath, "index.html");
    if (!File.Exists(filePath))
        return Results.NotFound();
    var html = await File.ReadAllTextAsync(filePath);
    // Inject <base> tag so relative paths (styles.css, app.js) resolve from "/"
    html = html.Replace("<head>", "<head>\n  <base href=\"/\">");
    return Results.Content(html, "text/html");
});
```

**Why `<base href="/">` and not modifying the source files:** The original `index.html` is used locally via `file://` protocol where relative paths work correctly. Changing to absolute paths (`/styles.css`) would break local file usage. The `<base>` injection at serve-time handles both cases.

#### 3. Frontend: Edit Mode Detection (app.js)
At startup, detect if the URL matches `/edit/{id}`:

```javascript
// Near the top of app.js, after SESSION_ID setup
function getEditModeFromUrl() {
    const match = window.location.pathname.match(/^\/edit\/([^/]+)$/);
    if (!match) return null;
    return { documentId: match[1] };

const EDIT_MODE = getEditModeFromUrl();
```

**Session ID for edit mode:** Use a deterministic session ID based on the document ID to ensure:
- Reloading the same edit URL reloads the same session
- Different edit documents don't collide
- Edit sessions don't collide with local sessions

#### 4. Frontend: Fetch & Load Document on Init
After initialization, if in edit mode, fetch and load the document:

```javascript
// After renderAll() and other init calls
if (EDIT_MODE) {
    loadEditDocument(EDIT_MODE.documentId);
}

async function loadEditDocument(docId) {
    const apiBase = window.location.origin;
    try {
        const res = await fetch(`${apiBase}/documents/${docId}`);
        if (!res.ok) {
            alert(`Failed to load document: ${res.status}`);
            return;
        }
        const data = await res.json();
        const normalized = normalizeState(data);
        await removeAllRouteCacheForSessionFromDb();
        replaceState(normalized);
        for (const city of state.cities || []) {
            await replaceRouteCacheForCityInDb(city.id, city.routeCache || []);
        }
        saveState();

        // Reset UI state (same as onImportJson)
        uiState.selectedDayId = "";
        uiState.selectedVerificationDayId = "";
        uiState.selectedDayIdsByCity = {};
        uiState.selectedVerificationDayIdsByCity = {};
        uiState.plannerDraft = null;
        uiState.plannerDraftCache = {};
        uiState.plannerDirty = false;
        uiState.lastValidation = null;
        uiState.routeInsights = null;

        resetResearcherForm();
        resetDayPlanForm();
        renderAll();
        renderRouteCacheStatus();
    } catch (err) {
        console.error(err);
        alert("Could not load shared document. Check your connection.");
    }
}
```

### Anti-Patterns to Avoid
- **Don't create a separate edit.html:** Reuse index.html — the whole point is loading the full app
- **Don't auto-save:** User explicitly decides when to push changes (locked decision)
- **Don't use innerHTML with API data:** Project convention is textContent/createElement (XSS prevention, per STATE.md decisions)
- **Don't duplicate normalizeState logic:** Reuse the exact same function — it's already defensive
- **Don't modify relative paths in source index.html:** Use `<base href="/">` injection at serve-time to keep local file:// usage working

## Common Pitfalls

### Pitfall 1: Relative Asset Paths Break Under Sub-Path
**What goes wrong:** `index.html` uses `href="styles.css"` and `src="app.js"`. When served from `/edit/abc-123`, browser resolves these to `/edit/styles.css` and `/edit/app.js` — 404.
**Why it happens:** Relative URLs are resolved against the current page's path.
**How to avoid:** Inject `<base href="/">` into the served HTML so all relative URLs resolve from root.
**Warning signs:** Unstyled page, JavaScript not loading, console 404 errors.

### Pitfall 2: CORS Self-Request
**What goes wrong:** API rejects fetch requests from the app served at the same origin.
**Why it happens:** Shouldn't happen — same-origin requests bypass CORS. But if `Cors:AllowedOrigins` doesn't include the API's own origin, some browsers may behave unexpectedly.
**How to avoid:** Verify same-origin fetch works. The current `.env.example` already includes `http://localhost:5000` in allowed origins, which matches the API host. Non-issue for same-origin.
**Warning signs:** CORS errors in browser console despite same-origin.

### Pitfall 3: Edit Session Overwrites Local Data
**What goes wrong:** Loading edit URL clobbers the user's local session data in localStorage.
**Why it happens:** Same `SESSION_ID` used for both local and edit mode.
**How to avoid:** Use a deterministic session ID like `edit_{documentId}` for edit mode, separate from local sessions.
**Warning signs:** User's local data disappears after visiting an edit link.

### Pitfall 4: Dockerfile COPY Context
**What goes wrong:** `COPY index.html .` fails because the Dockerfile's COPY context doesn't include the project root.
**Why it happens:** Docker build context is set to project root in `docker-compose.yml`, but Dockerfile working directory is `/src` (the API project dir).
**How to avoid:** Use the correct Dockerfile COPY paths relative to build context (project root). The existing Dockerfile already copies from `backend/ShareService.Api/` using the project root context — follow the same pattern.
**Warning signs:** Docker build failure, missing files in container.

### Pitfall 5: First Load Shows Empty State Briefly
**What goes wrong:** App initializes with empty localStorage state, renders empty UI, then async fetch replaces state — causing a visual flash.
**Why it happens:** `loadState()` runs synchronously at top of app.js, edit mode fetch is async.
**How to avoid:** Either accept the brief flash (simple), or show a loading indicator during fetch and hide the main UI. The viewer.html uses a "Loading document…" pattern that could be replicated.
**Warning signs:** UI flashes from empty to populated state.

### Pitfall 6: Upload Button Text Confusion
**What goes wrong:** Button says "Upload to API" but in edit mode it's actually updating an existing document.
**Why it happens:** UI doesn't adapt to edit mode context.
**How to avoid:** Optionally change button text to "Update Shared Document" or "Save to API" when in edit mode.
**Warning signs:** User confusion about whether they're creating a new document or updating.

```javascript
// Source: app.js lines 250-262
function getSessionIdFromUrl() {
    const params = new URLSearchParams(window.location.search);
    const value = String(params.get("session") || "").trim();
    if (value) return value;
    const sessionId = `session_${Date.now().toString(36)}_${Math.random().toString(16).slice(2, 8)}`;
    const nextUrl = new URL(window.location.href);
    nextUrl.searchParams.set("session", sessionId);
    window.history.replaceState({}, "", nextUrl.toString());
    return sessionId;
}
```

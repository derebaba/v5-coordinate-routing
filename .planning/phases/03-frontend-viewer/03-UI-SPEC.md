# Phase 3 — UI Design Contract

> Visual and interaction contract for the frontend share-URL display and the standalone viewer page. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | none |
| Preset | not applicable |
| Component library | none — vanilla HTML/CSS/JS |
| Icon library | none — no icons needed (text-only buttons) |
| Font | `"Segoe UI", Tahoma, Geneva, Verdana, sans-serif` (existing) |

**Rationale:** This project is a vanilla HTML/CSS/JS frontend with a .NET 8 backend. No React, no bundler, no npm. The existing `styles.css` uses CSS custom properties (`:root` variables) as the design system. The viewer page reuses the same design language to feel visually consistent without importing the full 920-line stylesheet. shadcn gate: not applicable (non-React project).

## Spacing Scale

Declared values derived from existing `styles.css` patterns. The project does NOT use a strict 8-point grid — it uses pragmatic values. The viewer page adopts the same rhythm:

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, chip internal padding, tight margins |
| sm | 8px | Compact element spacing, button padding-block, action bar gaps |
| md | 10px | Table cell padding, card internal padding, panel padding |
| lg | 14px | Panel padding (matches existing `.panel { padding: 14px }`), container gap |
| xl | 16px | Container horizontal padding, section vertical spacing |
| 2xl | 22px | Container top margin (matches existing `.container { margin: 22px 16px }`) |
| 3xl | 40px | Error state vertical padding, page bottom margin |

Exceptions:
- `6px` for tab gaps and small insets (existing pattern)
- `12px` for button horizontal padding and inline tool gaps (existing pattern)

| Body | 16px (1rem, browser default) | 400 | 1.5 |
| Label / Table cell | 14.9px (0.93rem) | 400 | 1.5 |

## Interaction Contracts

### 1. Upload → Share URL Display (SHARE-01, SHARE-02)

**Trigger:** Successful `POST /documents` response (res.ok === true).

**Behavior:**
1. Parse `url` field from JSON response (`data.url` — points to `/viewer/{id}`)
2. Clear `#upload-to-api-status` content (`status.innerHTML = ""`)
3. Create `<a>` element: `href` = share URL, `target` = `"_blank"`, `textContent` = share URL
4. Create `<button class="ghost">` with label "Copy Link"
5. Append link and button to `#upload-to-api-status`
6. Set `status.style.color = "var(--color-success, #27ae60)"`

**Copy button click:**
1. Call `navigator.clipboard.writeText(shareUrl)` (async, in try/catch)
2. On success: change button text to "Copied!", revert after 2000ms
3. On failure: change button text to "Copy failed"

**Fallback:** If `data.url` is empty/missing, show plain text: "Uploaded successfully."

### 2. Viewer Page Load (VIEW-01, VIEW-02)

**Trigger:** Browser navigates to `/viewer/{id}`.

**Behavior:**
1. Extract `id` from `window.location.pathname.split('/').pop()`
2. Show `#loading` ("Loading document…")
3. Fetch `GET /documents/{id}` (relative path — same origin)
4. **On 200:** Hide `#loading`, show `#content` + `#actions`, call `renderDocument(data)`
5. **On 404:** Hide `#loading`, show `#error` with error message
6. **On network error:** Hide `#loading`, show `#error` with connection error message
7. **On malformed data:** Show `#error` with format error message
8. **After successful load:** Update `document.title` to include first city name

### 3. Download Button (VIEW-03)

**Trigger:** Click "Download JSON" button on viewer page.

**Behavior:**
1. Create `Blob` from stored JSON data: `new Blob([JSON.stringify(data, null, 2)], { type: "application/json" })`
2. Create object URL and invisible `<a>` element
3. Set `download` attribute to `shared-document-{id}.json`
4. Trigger click, then revoke object URL

## Accessibility

| Concern | Implementation |
|---------|---------------|
| Share URL link | Must be a real `<a>` tag with `href` — keyboard-navigable, screen-reader accessible |
| Copy Link button | Must be a `<button>` element (not div/span) for keyboard activation |
| Download button | Must be a `<button>` element with descriptive text |
| Error messages | Use `role="alert"` on `#error` element so screen readers announce errors |
| Table semantics | Use proper `<table>`, `<thead>`, `<th>`, `<tbody>`, `<td>` elements |
| Color contrast | `--text` (#1d2a1f) on `--panel` (#ffffff) = 15.3:1 ratio ✓ |
| Color contrast | `--muted` (#5d6b60) on `--panel` (#ffffff) = 5.1:1 ratio ✓ |
| Color contrast | `--primary` (#1e7a46) on `--panel` (#ffffff) = 5.8:1 ratio ✓ |
| Focus styles | Inherit existing `input:focus` pattern: `outline: 2px solid #9cc9a8; outline-offset: 1px` — apply to buttons too |

## Security Contracts

| Concern | Rule |
|---------|------|
| XSS prevention | NEVER use `innerHTML` with data from the API. Always use `textContent` or `createElement` for rendering document data (school names, notes, city names). |
| URL construction | Share URL comes from the API response `data.url`. Do not construct URLs from user input. |
| Same-origin fetch | Viewer fetches `/documents/{id}` using a relative path — no cross-origin requests. |
| Input sanitization | The `{id}` extracted from the URL path is only used in the fetch URL — never inserted into HTML. |
